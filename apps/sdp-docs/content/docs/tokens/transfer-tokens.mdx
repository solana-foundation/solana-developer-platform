---
title: Transfer Tokens
description: Send tokens between accounts using SDP's payment orchestration.
---

SDP handles token transfers through its Payments API. The source wallet must be configured in SDP with signing enabled.

## Prerequisites

- A deployed token with minted supply
- A wallet with signing configured as the transfer source
- `payments:write` permission on the API key

## How it works

<HowItWorks>

<Step number={1} title="Set up the transfer">

Specify the source, destination, token, and amount. Always include an idempotency key to protect against duplicate transfers on network errors.

<StepPanel>
<Tabs items={["UI", "API"]} groupId="how-it-works">
<Tab value="UI">

Go to **Payments** in the sidebar and click **New transfer**.

Fill in:

- **Source** — a wallet you control in SDP
- **Destination** — any valid Solana wallet address
- **Token** — select from your deployed tokens
- **Amount** — human-readable units (e.g., `0.5`)
- **Memo** — optional note recorded onchain

</Tab>
<Tab value="API">

Prepare the request body:

```typescript
const body = {
  source: "3xYZa...2aBc",       // your controlled wallet
  destination: "7xKXz...9fGh",  // recipient wallet
  token: "9aBCd...4eEf",        // onchain mint address
  amount: "500000",              // smallest unit (6 decimals = 0.5 tokens)
  memo: "Payment for invoice #1234",
};
```

`token` is the mint address returned by the deploy step, not the SDP token ID.

</Tab>
</Tabs>
</StepPanel>

</Step>

<Step number={2} title="Execute the transfer">

Submit the transfer. In execute mode SDP signs and submits. In prepare mode you receive an unsigned transaction.

<StepPanel>
<Tabs items={["UI", "API"]} groupId="how-it-works">
<Tab value="UI">

Click **Review** to see the transfer summary, then **Confirm** to submit.

The transfer status updates to `Confirmed` once the transaction is finalized.

</Tab>
<Tab value="API">

**Execute mode:**

```bash
curl -X POST https://api.solana.com/v1/payments/transfers \
  -H "Authorization: Bearer sk_test_..." \
  -H "Content-Type: application/json" \
  -H "Idempotency-Key: transfer-001" \
  -d '{
    "source": "3xYZa...2aBc",
    "destination": "7xKXz...9fGh",
    "token": "9aBCd...4eEf",
    "amount": "500000",
    "memo": "Payment for invoice #1234"
  }'
```

```typescript
const res = await fetch("https://api.solana.com/v1/payments/transfers", {
  method: "POST",
  headers: {
    Authorization: "Bearer sk_test_...",
    "Content-Type": "application/json",
    "Idempotency-Key": "transfer-001",
  },
  body: JSON.stringify(body),
});
const { data } = await res.json();
// data.transfer.id     — for status polling
// data.transfer.status — "pending" | "confirmed" | "failed"
```

**Prepare mode** — append `/prepare` and optionally add `"options": { "simulate": true }`.

See [Prepare vs Execute](/docs/tokens/prepare-vs-execute) for guidance.

</Tab>
</Tabs>
</StepPanel>

</Step>

<Step number={3} title="Verify the transfer">

Confirm the transfer completed and balances updated as expected.

<StepPanel>
<Tabs items={["UI", "API"]} groupId="how-it-works">
<Tab value="UI">

The transfer appears in **Payments** history with a `Confirmed` status and an onchain transaction signature.

Click the record to see full details including the Solana Explorer link.

</Tab>
<Tab value="API">

```typescript
// data.transfer.status    — "confirmed"
// data.transfer.signature — onchain tx signature
```

Poll status for async confirmations:

```bash
curl https://api.solana.com/v1/payments/transfers/txn_abc123 \
  -H "Authorization: Bearer sk_test_..."
```

</Tab>
</Tabs>

## Idempotency

Always include an `Idempotency-Key` header. Retrying with the same key on network errors ensures the transfer executes only once.
