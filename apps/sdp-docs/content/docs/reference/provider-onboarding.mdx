SDP currently supports infrastructure provider integrations across custody, RPC, compliance, and ramps. We will soon add future integration categories as SDP expands. The onboarding process is self-service for the initial contribution: select the matching category, complete the matching intake when available, give the Solana Foundation team access to validate the integration, and submit the work for review.

Onboarded infrastructure providers are presented as available integrations, not endorsed or recommended partners. Institutions using SDP are responsible for conducting their own due diligence on any provider they select.

## Onboarding steps

1. Review the criteria PDF for your infrastructure category.
2. Complete the matching category intake form when it is available.
3. Provide sandbox or playground API access for the Solana Foundation team.
4. Share provider documentation, test credentials, supported networks, rate limits, and any known sandbox limitations.
5. Build the SDP integration with tests and docs.
6. Open a pull request that links to the intake submission or tracking issue.

The SDP maintainers review the intake for completeness within five business days. Gate evaluation usually takes one to three weeks depending on category, applicant responsiveness, documentation completeness, and whether engineering benchmarks or devnet validation are required.

## Access requirements

Infrastructure providers must give the Solana Foundation team API access to a sandbox or playground environment. Access should be suitable for repeatable testing and validation, including local development and automated checks where possible.

At minimum, include:

- Sandbox API keys or equivalent credentials.
- Test accounts, wallets, organization IDs, endpoint URLs, or settlement rails when required.
- Supported networks, such as Solana devnet and mainnet-beta.
- Known rate limits, account limits, feature flags, and sandbox limitations.
- Contact information for technical support during review.

## Evaluation criteria

The SDP team evaluates the finalized criteria for the relevant category. Every category uses hard gates before scored evaluation. Providers that do not clear the gates may still be tracked for future support, but they should not be presented as generally available in SDP until the gaps are resolved.

### Custodial enterprise wallet criteria

[Download the custodial enterprise wallet criteria PDF](/provider-onboarding/custodial-wallet-vendor.pdf).

> "Hard Gates are binary pass/fail requirements. Failure on any gate disqualifies the applicant."

Custodial enterprise wallet providers are evaluated for private-key custody and signing on behalf of institutional clients. SDP integration readiness also includes compatibility with the Solana keychain interface before the provider is wired into SDP.

Hard gates:

- Solana technical capability: arbitrary transaction signing, Token-2022 transaction signing, and correct Solana account-model handling.
- Security baseline: institutional-grade key management, recent external penetration testing, and a documented incident response plan.
- Regulatory standing: custodian, trust company, or equivalent licensing or registration plus an AML/KYC program.
- Operational viability: funded operating business and at least one live institutional client in production.

Scored evaluation weights:

- Technical depth: 50%.
- Security and compliance maturity: 30%.
- Operational maturity: 20%.

The custody intake should cover company overview, regulatory standing, Solana technical capabilities, key management architecture, and client references.

### RPC infrastructure criteria

[Download the RPC infrastructure criteria PDF](/provider-onboarding/rpc-providers.pdf).

> "Because every provider in the pool receives live production traffic, a degraded provider directly impacts SDP reliability."

RPC infrastructure providers are evaluated for standard Solana JSON-RPC endpoints used by SDP for transaction submission and read operations. Enhanced services such as geyser/gRPC streaming, DAS, or custom APIs are scored differentiators, but they are not required for onboarding.

Hard gates:

- Solana RPC capability: full JSON-RPC compliance, transaction landing reliability, devnet and mainnet-beta support, and a latency baseline under normal conditions.
- Operational baseline: uptime track record, clear rate limits and capacity, and incident response communication.
- Business viability: at least six months of production Solana RPC service and no disqualifying operational or legal events.

Scored evaluation weights:

- Performance and reliability: 50%.
- Solana ecosystem depth: 25%.
- Operational and business maturity: 25%.

The RPC intake should cover company overview, Solana RPC capabilities, infrastructure architecture, uptime history, rate limits, incident process, and enhanced service offerings.

### Ramp criteria

[Download the ramp criteria PDF](/provider-onboarding/ramp-criteria.pdf).

> "Ramp providers enable the conversion between fiat currencies and digital assets on Solana."

Ramp providers are evaluated across institutional ramp, retail onramp, and issuer-as-ramp profiles. The SDP ramp integration layer is still being defined, so the assessment weighs coverage and compliance maturity heavily while technical integration details are finalized during onboarding.

Hard gates:

- Solana capability: settlement to Solana addresses or Solana mint/burn flows, plus stablecoin support.
- Regulatory and compliance baseline: relevant licensing or registration, AML/KYC/KYB program, and sanctions screening.
- Operational viability: production ramp service, active banking relationships, and no disqualifying events.

Scored evaluation weights:

- Coverage and reach: 40%.
- Compliance and risk maturity: 35%.
- Operational maturity: 25%.

The ramp intake should cover company overview, provider type classification, fiat and stablecoin coverage, payment rail integrations, regulatory licenses, geographic reach, KYC/KYB flows, settlement capabilities, and API readiness.

## Pull request expectations

An SDP infrastructure provider contribution usually needs:

- Provider metadata and capability registration.
- API support for initialization, provider-specific configuration, and the category's runtime operations.
- Dashboard support for setup, status, and category-specific workflows when applicable.
- Tests for successful setup, failure states, disabled configuration, and missing credentials.
- User-facing docs for setup requirements, sandbox behavior, and provider-specific limits.

Open the pull request only after the matching intake is complete, or after the SDP team has created a tracking issue while intake forms are still being finalized, and enough sandbox access is available for validation.
